# 服务器数据丢失事故分析报告

**日期**: 2026-04-02
**服务器**: ubuntu-s-2vcpu-4gb-120gb-intel-nyc3-01 (167.99.115.47)
**项目**: BYCIGAR 雪茄电商平台

---

## 一、用户反馈的现象
- 前端页面可以打开，但**数据显示不出来**（商品列表、分类、横幅等为空）
- **创建用户失败**（注册功能不可用）
- 服务器部署在远程，不确定原因，认为"宕机"

---

## 二、排查过程与关键发现

### 1. 初步检查系统资源（排除基础设施故障）
- `top`：CPU 空闲 97.2%，内存使用 29.3%（共 4GB），负载极低 → **无资源瓶颈**
- `df -h`：磁盘使用仅 7% → **无磁盘满问题**
- `/var/log/syslog`：无 OOM、内核 panic 等错误 → **系统层面正常**

### 2. 检查容器状态
- `docker ps -a` 显示所有容器均在运行：
  - `bycigar-backend` (后端) – Up 5 hours
  - `bycigar-mysql` (MySQL) – Up 31 minutes (healthy)
  - `bycigar-frontend` (前端) – Up 5 hours
  - `bycigar-minio` (对象存储) – Up 6 hours
  - `bycigar-certbot` – Up 5 hours
- MySQL 容器运行时间明显短于其他容器，说明中间重启过。

### 3. 查看后端日志
执行 `docker logs bycigar-backend --tail 100`，发现大量重复错误：
```
Error 1049 (42000): Unknown database 'bycigar'
```

### 4. 查看 MySQL 容器日志（关键线索）
执行 `docker logs bycigar-mysql --tail 200`，发现：

**03:23~03:24 期间，MySQL 反复崩溃重启约 10 次**，每次报错：
```
[ERROR] [MY-000068] [Server] unknown option '--skip-character-set-client-handshake'.
[ERROR] [MY-010119] [Server] Aborting
```
最终在 03:24:32 成功启动（可能偶然跳过了该参数）。

**08:24:32 收到 SHUTDOWN 命令**（来自 root 用户），08:24:37 重新启动成功。

### 5. 验证 Docker Volume
```
docker volume inspect mysql_data
[]
Error response from daemon: get mysql_data: no such volume
```
**服务器上根本不存在 `mysql_data` 这个 Docker Volume**，尽管 compose 文件中定义了它。

### 6. 验证 Compose 配置合并
```
docker inspect bycigar-mysql | grep config_files
"com.docker.compose.project.config_files": "/opt/bycigar/docker-compose.yml,/opt/bycigar/docker-compose.prod.yml"
```
两个文件合并使用，环境变量 `MYSQL_DATABASE=bycigar` 存在。

### 7. 发现可疑数据库
在排查过程中发现数据库列表中存在 `RECOVER_YOUR_DATA` 数据库，这是典型的勒索软件特征。

---

## 三、根本原因（两个叠加）

### 根因 1：`docker-compose.prod.yml` 覆盖了 volumes 定义，导致 MySQL 数据没有持久化

部署命令为 `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`。

- `docker-compose.yml` 底部定义了 `mysql_data` 和 `minio_data` 两个 named volume
- `docker-compose.prod.yml` 底部只定义了 `certbot-webroot`
- **Docker Compose 合并多个文件时，`volumes:` 顶层键会被后者替换，而非合并**
- 结果：`mysql_data` 和 `minio_data` 从未被创建，MySQL 数据存储在容器可写层
- 容器重建时数据全部丢失

### 根因 2：`my.cnf` 中 `skip-character-set-client-handshake` 在 MySQL 8.4 中已被移除

`mysql/my.cnf` 包含 `skip-character-set-client-handshake`，该选项在 MySQL 8.4 中不被识别。这导致 MySQL 启动时立即崩溃。Docker 的 `restart: unless-stopped` 策略不断重启，形成启动-崩溃循环。

---

## 四、事故时间线还原

1. **初始部署** — MySQL 容器首次启动时，因某种原因跳过了无效参数（或之前用的是旧版 MySQL），`MYSQL_DATABASE=bycigar` 自动建库，用户手动导入测试数据，一切正常
2. **03:23** — MySQL 容器因某种原因重启（可能是 Docker 守护进程重启或宿主机维护），`skip-character-set-client-handshake` 导致反复崩溃约 10 次
3. **03:24:32** — MySQL 最终成功启动，但此时因为没有持久化卷，启动的是全新空数据库
4. **08:24:32** — 收到 SHUTDOWN 命令再次重启，数据仍然为空
5. **发现故障** — 后端报 `Unknown database 'bycigar'`，用户手动建库，GORM AutoMigrate 自动创建空表，但业务数据已永久丢失

---

## 五、修复步骤

### 修复 1：`my.cnf` 移除不兼容参数
```diff
  [mysqld]
  character-set-server = utf8mb4
  collation-server = utf8mb4_unicode_ci
  init_connect = 'SET NAMES utf8mb4'
- skip-character-set-client-handshake
```
`init_connect` + `character-set-server` 已足够确保 utf8mb4，无需该选项。

### 修复 2：`docker-compose.prod.yml` 补全 volumes 定义
```diff
  volumes:
+   mysql_data:
+   minio_data:
    certbot-webroot:
```
确保合并后 `mysql_data` 和 `minio_data` 都存在。

### 修复 3：my.cnf 权限修复
```bash
chmod 644 /opt/bycigar/mysql/my.cnf
docker compose restart mysql
```

### 修复 4：重新部署
```bash
cd /opt/bycigar
docker compose -f docker-compose.yml -f docker-compose.prod.yml down
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

### 修复 5：验证持久化卷
```bash
docker volume ls | grep bycigar
# 应显示:
# bycigar_mysql_data
# bycigar_minio_data
# bycigar_certbot-webroot
```

## 七、安全加固

### 7.1 发现的安全风险

在排查过程中发现可疑数据库 `RECOVER_YOUR_DATA`，这是典型的勒索软件特征。

**可能原因**:
- MySQL 曾短暂暴露在公网
- 弱密码被暴力破解
- 重新部署后已清除

### 7.2 安全加固措施

#### 7.2.1 防火墙配置 (UFW)
```bash
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp   # SSH
ufw allow 80/tcp   # HTTP
ufw allow 443/tcp  # HTTPS
ufw enable
```

**验证结果**:
```
Status: active

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW       Anywhere
80/tcp                     ALLOW       Anywhere
443/tcp                    ALLOW       Anywhere
```

#### 7.2.2 MySQL 端口隔离
确保 MySQL 3306 端口不对外暴露，只在 Docker 内部网络通信。

验证命令：
```bash
ss -tlnp | grep 3306
# 应显示: 127.0.0.1:3306 或无输出
```

#### 7.2.3 可疑数据库清理
在 `server-setup.sh --security-only` 中自动检测并删除 `RECOVER_YOUR_DATA` 数据库。

#### 7.2.4 自动备份
创建 `backup.sh` 脚本，每日凌晨 3:00 自动备份数据库：

```bash
# 设置定时任务
crontab -e
# 添加:
0 3 * * * /opt/bycigar/backup.sh >> /var/log/bycigar-backup.log 2>&1
```

备份策略：
- 存储位置: `/opt/backups/`
- 保留天数: 7 天
- 文件格式: `bycigar_YYYYMMDD_HHMMSS.sql.gz`

**验证结果**:
```
-rw-r--r-- 1 root root 20K Apr 2 10:07 /opt/backups/bycigar_20260402_100723.sql.gz
```

## 十、经验教训

1. **多文件合并时注意覆盖问题**

   - 升级前检查配置变更

## 十一、代码层面的潜在风险（已完成修复）

以下问题在 **2026-04-02** 完成修复：

### 严重 - 会导致进程直接崩溃 ✅ 已修复
1. **WebSocket goroutine 没有 panic recovery**（`ws/hub.go` WritePump/ReadPump）→ 添加 defer recover
2. **6个后台清理 goroutine 没有 recover()**（chat/notification/image/soft_delete/order/cart cleanup）→ 添加 panic recovery
3. **邮件发送 goroutine 没有 recover()**（`admin_order.go`）→ 添加 panic recovery
4. **完全没有 graceful shutdown**（`main.go` 无 signal handling）→ 添加 SIGINT/SIGTERM 处理 + 10s 超时

### 高危 - 特定条件下可能 panic ✅ 已修复
5. **17处不安全的类型断言** `userID.(uint)` 无 comma-ok 检查 → 改为 comma-ok 模式
6. **`loginFailures` map 无上限增长**（被机器人攻击时内存泄漏）→ 添加 10000 条上限 + 30分钟后台清理
7. **`ExportAdminOrders` 一次加载所有订单到内存** → 添加 10000 条记录上限

### 中等 - 运维风险 ✅ 已修复
8. **没有 `/health` 健康检查端点** → 添加 `GET /health` 返回 `{"status":"ok"}`

### 修改的文件清单

| 文件 | 修改内容 |
|------|----------|
| `cmd/main.go` | graceful shutdown + /health 端点 |
| `internal/ws/hub.go` | panic recovery + Shutdown() 方法 |
| `pkg/utils/*_cleanup.go` (6个) | panic recovery |
| `internal/handlers/admin_order.go` | 邮件 panic recovery + 导出限制 |
| `internal/handlers/auth.go` | loginFailures 上限 + 后台清理 |

## 十二、关键经验总结

### 12.1 MySQL 8.4 兼容性
- **`skip-character-set-client-handshake` 已被移除**，需从 my.cnf 删除
- 使用 `init_connect = 'SET NAMES utf8mb4'` + `character-set-server = utf8mb4` 确保字符集

### 12.2 数据导入字符集
- **导入数据必须指定字符集**：`mysql --default-character-set=utf8mb4`，否则中文会乱码

### 12.3 安全硬性规定
- **数据库端口严禁暴露公网**，必须使用 UFW 限制
- **必须使用强密码**，弱密码会被暴力破解
- **定期检查可疑数据库**（如 `RECOVER_YOUR_DATA` 是勒索软件特征）

### 12.4 备份策略
- **必须配置自动备份**，每日至少一次
- 备份文件保留 7 天以上
- 定期验证备份可恢复

### 12.5 Docker Compose 多文件合并
- `volumes`、`networks` 顶级字段会被**覆盖**而非合并
- 生产文件必须完整定义所有需要的资源

### 12.6 代码健壮性
- **所有 goroutine 必须有 panic recovery**
- **类型断言必须使用 comma-ok 模式**
- **map 必须有上限和清理机制**
- **服务必须有 graceful shutdown**
